1. VISÃO GERAL DO PRODUTO
================================================================================
Nome do produto : "Call Intelligence — Sales Coaching IA"
Pasta/projeto   : AdBalanceBilling (nome herdado da pasta; o produto é de análise
                  de calls comerciais, NÃO de billing).

O QUE É:
Plataforma de Sales Coaching / Call Intelligence para times comerciais. O ativo
central é a CALL do Closer (vendedor). O sistema transforma a transcrição bruta
de uma call em:
  - Análise estruturada por IA (notas por critério comercial + textos)
  - Feedback final prático e acionável (não genérico)
  - Diagnóstico de Causa Raiz comercial (skill "Prisma")
  - Relatório Comercial da Oportunidade (visão do cliente + chance de venda)
  - Resumo automático enviado ao Slack
  - Dashboard de evolução do Closer ao longo do tempo

Setor-alvo da skill comercial: MATERIAIS DE CONSTRUÇÃO (varejo/atacado,
distribuidoras de ferro e aço, lojas físicas, distribuidores regionais).

FLUXO PRINCIPAL:
1. Recebe/importa a transcrição de uma call (manual, upload de arquivo, ou
   webhook do Tactiq).
2. IA (Google Gemini) analisa a call.
3. IA identifica pontos fortes/fracos, oportunidades, próximos passos, causa raiz
   e relatório comercial.
4. Sistema gera resumo estruturado e salva tudo no banco.
5. Sistema envia resumo no Slack (se configurado).
6. Dashboard registra e mostra evolução.
7. Feedback final com nota, observações e recomendações práticas.

6. BANCO DE DADOS (Supabase / Postgres 17)
================================================================================
PRINCÍPIO DE DESIGN:
- scores em tabela própria (1 linha por critério por análise) -> permite
  comparação histórica e mudar a rubrica sem migração.
- Cada análise guarda model + rubric_version -> notas comparáveis no tempo.
- O JSON bruto da IA é salvo em call_analyses.raw (jsonb). Os blocos root_cause e
  relatorio_comercial são lidos de raw (NÃO têm coluna própria).
- RLS LIGADO em todas as tabelas, SEM políticas públicas. Browser nunca consulta
  o banco direto; só o servidor (service_role, que bypassa RLS).

ENUMS:
  call_status     : recebida | pendente | em_analise | concluida | revisada |
                    erro_na_analise
  analysis_status : pendente | em_analise | concluida | falhou
  action_status   : pendente | em_andamento | aplicada | descartada
  call_source     : transcricao | link | audio | video | tactiq

FUNÇÃO/TRIGGER:
  set_updated_at() -> trigger BEFORE UPDATE que seta updated_at = now().
  Aplicado em: closers, calls, call_analyses, improvement_actions.
  (Use "create or replace trigger" para idempotência — Postgres 14+.)

TABELAS (campo a campo):

closers
  id uuid PK default gen_random_uuid()
  name text NOT NULL
  email text
  avatar_url text
  role text default 'Closer'
  active boolean NOT NULL default true
  created_at timestamptz default now()
  updated_at timestamptz default now()  (trigger)

criteria   (rubrica de avaliação — reference data)
  id uuid PK
  key text UNIQUE NOT NULL          (ex.: 'diagnostico')
  label text NOT NULL               (rótulo pt-BR)
  description text                  (o que é "bom")
  weight numeric(4,2) default 1.0
  sort_order int default 0
  active boolean default true
  version int default 1
  created_at timestamptz default now()

calls
  id uuid PK
  closer_id uuid NOT NULL REFERENCES closers(id) ON DELETE CASCADE
  client_name text NOT NULL         (cliente/lead)
  call_date date NOT NULL default current_date
  source call_source default 'transcricao'
  recording_url text                (link da gravação)
  transcript text                   (transcrição em texto)
  status call_status default 'pendente'
  duration_sec int
  created_at / updated_at timestamptz (trigger)
  índices: closer_id, call_date, status

call_analyses
  id uuid PK
  call_id uuid NOT NULL REFERENCES calls(id) ON DELETE CASCADE
  status analysis_status default 'pendente'
  overall_score numeric(4,2)        (0..10)
  summary text
  closer_talk_pct int               (% fala do closer)
  client_talk_pct int               (% fala do cliente)
  model text                        (modelo de IA usado)
  rubric_version int default 1
  raw jsonb                         (payload completo da IA — inclui root_cause e
                                     relatorio_comercial)
  error text
  created_at / updated_at timestamptz (trigger)
  índice: call_id

scores
  id uuid PK
  analysis_id uuid NOT NULL REFERENCES call_analyses(id) ON DELETE CASCADE
  criterion_key text NOT NULL
  score numeric(4,2) NOT NULL       (0..10)
  justification text
  created_at timestamptz
  índices: analysis_id, criterion_key

call_highlights   (momentos importantes)
  id uuid PK
  analysis_id uuid NOT NULL REFERENCES call_analyses(id) ON DELETE CASCADE
  kind text default 'momento'       (dor|objecao|virada|risco|momento)
  timestamp_ref text                (ex.: "12:30")
  quote text
  comment text
  created_at timestamptz

feedbacks
  id uuid PK
  analysis_id uuid NOT NULL REFERENCES call_analyses(id) ON DELETE CASCADE
  strengths text[]                  (pontos fortes)
  weaknesses text[]                 (pontos fracos)
  keep_doing text[]                 (o que manter)
  fix_doing text[]                  (o que corrigir)
  final_comment text                (comentário final humano)
  better_approach text              (como conduzir melhor, com exemplos)
  created_at timestamptz

improvement_actions
  id uuid PK
  analysis_id uuid NOT NULL REFERENCES call_analyses(id) ON DELETE CASCADE
  closer_id uuid NOT NULL REFERENCES closers(id) ON DELETE CASCADE
  title text NOT NULL
  detail text
  priority int default 2            (1 alta, 2 média, 3 baixa)
  status action_status default 'pendente'  ('aplicada' alimenta "feedbacks aplicados")
  created_at / updated_at timestamptz (trigger)
  índices: closer_id, status

slack_logs
  id uuid PK
  call_id uuid REFERENCES calls(id) ON DELETE SET NULL
  channel text
  payload jsonb
  ok boolean default false
  error text
  created_at timestamptz

webhook_logs   (migration 0003 — auditoria de webhooks recebidos)
  id uuid PK
  source text default 'tactiq'
  call_id uuid REFERENCES calls(id) ON DELETE SET NULL
  ok boolean default false
  status text
  error text
  payload jsonb
  created_at timestamptz
  índices: source, call_id

  Header: x-goog-api-key: <GEMINI_API_KEY>, Content-Type: application/json
- Modelo: gemini-2.5-flash (leve/free; 2.0-flash tinha cota 0 nesse projeto).

  dashboard, Slack) e a análise fica desabilitada com aviso claro.

13. SKILL COMERCIAL "PRISMA" (Análise de Causa Raiz)
================================================================================
Arquivo: web/src/lib/ai/skills/prisma-rca.ts (constante PRISMA_RCA_SKILL,
injetada no SYSTEM_PROMPT). Persona: diretor comercial sênior, causa raiz para
o setor de MATERIAIS DE CONSTRUÇÃO. Pontos-chave do conteúdo:

- Contexto/ICP: varejo/atacado de materiais, ferro e aço, lojas físicas,
  distribuidores. ICP = empresa com operação real (loja, equipe, estoque,
  região, potencial). NÃO ICP = prestador, construtora, autônomo, MEI sem
  estrutura. Objetivo: anúncios -> demanda qualificada -> REUNIÃO. WhatsApp NÃO
  fecha venda, NÃO passa preço, NÃO explica tudo — serve para avançar p/ reunião.
- Ciclo: anúncio -> lead -> 1º contato -> qualificação -> agendamento ->
  confirmação -> diagnóstico -> valor -> solução -> objeções -> avanço ->
  fechamento/follow-up.
- Comunicação firme/madura. EVITAR frases fracas ("só passando pra ver", "tem
  interesse?", "qualquer coisa me chama", "fico à disposição", "sem compromisso",
  "talvez", "acho que"). PREFERIR ("O ponto é o seguinte", "Faz sentido
  alinharmos numa reunião", "O melhor próximo passo é").
- Straight Line Selling: 3 certezas (solução, vendedor, empresa); avaliar qual
  faltou; ver se conduziu em linha reta ou abriu desvios.
- SPIN: Situação, Problema, Implicação, Necessidade. Dor só citada = decisão
  fraca; dor AMPLIADA = urgência.
- Objeções: resistência (não rejeição). Estrutura: reconhecer -> validar ->
  isolar -> real ou fuga -> retomar valor -> reduzir risco -> microcompromisso ->
  avançar. Tipos: prospecção, red herrings, microcompromisso, compra.
- 20 categorias de causa: ICP, canal, anúncio, oferta, posicionamento, abordagem,
  tempo de resposta, qualificação, WhatsApp, ligação, reunião, construção de
  valor, confiança, autoridade, urgência, preço, follow-up, processo, emocional,
  estratégica.
- Regras: não inventar dados; nada genérico; não culpar só lead nem só vendedor;
  buscar causa mais CONTROLÁVEL; não sugerir desconto/preço no WhatsApp/explicação
  longa/insistência vazia; follow-up sempre com contexto+motivo+direção.
- Lógica: sem diagnóstico não há venda consultiva; sem dor não há urgência; sem
  autoridade não há decisão; sem confiança não há avanço; sem microcompromisso o
  pipeline trava; sem próximo passo o lead esfria; sem controle o lead conduz;
  sem valor o preço vira objeção; sem processo o comercial depende de sorte.
- Saída: separa sintoma/causa aparente/causa raiz; aponta etapa que quebrou,
  comportamento ideal, correção imediata, prevenção, e (quando faz sentido) uma
  mensagem/fala pronta no tom firme; termina com próximo passo objetivo.

RENDERIZAÇÃO: RootCausePanel (na página de feedback). Badges de gravidade/
confiança/etapa; diagnóstico direto em destaque; sintoma->aparente->raiz; causas
secundárias + categorias; correção (deveria/corrigir/prevenir); mensagem
recomendada; próximo passo.

16. INTEGRAÇÃO TACTIQ (Webhook)
================================================================================
Rota: POST /api/webhooks/tactiq  (web/src/app/api/webhooks/tactiq/route.ts)
URL local      : http://localhost:3000/api/webhooks/tactiq
URL produção   : NEXT_PUBLIC_APP_URL + /api/webhooks/tactiq
Tela           : Configurações -> card "Webhook do Tactiq" (CopyField).

Payload (JSON):
  titulo_da_call, data_da_call, closer, cliente, empresa, link_da_reuniao,
  transcricao (OBRIGATÓRIO, >=40 chars), participantes (string|array),
  origem ("tactiq")

Fluxo:
  1. (Opcional) Segredo: se TACTIQ_WEBHOOK_SECRET setado, exige header
     x-webhook-secret (ou ?secret=); senão 401.
  2. Valida transcrição (400 amigável se faltar).
  3. Resolve closer pelo nome (ilike) ou cria; sem nome -> "Não informado".
  4. client_name = "cliente — empresa"; prepende contexto (título/empresa/
     participantes) à transcrição p/ enriquecer a análise.
  5. Insere call: source 'tactiq', status 'recebida'. Loga em webhook_logs (ok).
  6. Se !isAIEnabled() -> mantém 'recebida' e retorna (modo manual).
  7. runAnalysis(callId): salva análise + Slack.
  8. Se análise falhar -> call status 'erro_na_analise' + webhook_logs(error).
  Respostas: 200 {ok,call_id,status,message} | 400 validação | 401 segredo |
             500 erro ao salvar.
  GET de cortesia retorna info do serviço.

Conectar Tactiq -> usar Zapier ou Make como ponte:
  Zapier: Trigger Tactiq "New Meeting Transcript" -> Action "Webhooks by Zapier"
    POST (JSON) para a URL; mapear campos; header x-webhook-secret se usar segredo.
  Make: módulo Tactiq -> HTTP POST (Raw/JSON) p/ a URL.
  Em dev local, expor com túnel (ex.: ngrok) para o Zapier/Make alcançarem.

TESTE REALIZADO (ponta a ponta, OK):
  POST real -> call salva (source tactiq, status concluida) -> análise Gemini
  (gemini-2.5-flash, 12 notas, feedback, root_cause, relatorio_comercial) ->
  webhook_logs criado (ok, 'recebida') -> sem erros no terminal.
